# 网站修复与诊断报告

## 1. 问题诊断 (Diagnosis)

在检查过程中，我们发现了以下导致网站无法正常运行的关键问题：

### 1.1 配置文件编码错误
- **症状**: 后端服务器启动时无法读取环境变量，提示 `MONGODB_URI` 未定义，尽管 `.env` 文件中存在该配置。
- **原因**: `.env` 文件使用了 **UTF-16LE** 编码格式，而 Node.js 的 `dotenv` 库默认仅支持 UTF-8 编码，导致解析失败。

### 1.2 端口冲突
- **症状**: 后端服务启动失败，报错 `EADDRINUSE: address already in use 0.0.0.0:5000`。
- **原因**: 默认端口 `5000` 被系统进程 `GCPHost.exe` 占用。

### 1.3 数据库连接失败
- **症状**: 服务器启动时崩溃，报错 `querySrv ENOTFOUND _mongodb._tcp.cluster0.qkwys0c.mongodb.net`。
- **原因**: 配置的远程 MongoDB Atlas 地址无法通过 DNS 解析（可能是网络问题或实例已失效），导致连接超时。

## 2. 实施修复 (Applied Fixes)

针对上述问题，我们执行了以下修复操作：

### 2.1 转换文件编码
- **操作**: 编写并运行了脚本，将 `.env` 文件的编码格式从 UTF-16LE 转换为 **UTF-8**。
- **结果**: 环境变量现在可以被系统正确读取。

### 2.2 端口重新配置
- **操作**: 将 `.env` 中的后端端口配置 `PORT` 从 `5000` 修改为 **`5001`**。
- **结果**: 避开了端口冲突。前端代理 (`setupProxy.js`) 原本就配置指向 5001 端口，因此无需额外修改前端配置。

### 2.3 增加数据库容错机制
- **操作**: 修改了 `src/server/server.js`，增加了数据库连接的 **Fallback（降级）机制**。
- **逻辑**: 当连接远程 MongoDB 失败时，系统会自动尝试启动 **内存数据库 (In-Memory MongoDB)** 作为备用。
- **结果**: 即使没有互联网连接或远程数据库失效，项目也能在本地正常启动和演示。
